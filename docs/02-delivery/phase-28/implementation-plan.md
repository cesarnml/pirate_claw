# Phase 28 Implementation Plan

Phase 28 adds the human security boundary required before Pirate Claw can be treated as a release-ready NAS appliance. A single local owner account, daemon-owned durable auth state, web-issued JWT sessions, and web-enforced route guards replace the unauthenticated control surface.

## Grill-Me decisions locked for this phase

- **JWT signed with daemon-owned secret.** Web reads `config/auth/session-secret` from the shared volume at startup — same pattern as the daemon write token in `hooks.server.ts`. No daemon round-trip for session validation. 30-day absolute expiry, `httpOnly + SameSite=Strict` cookie.
- **First-run takeover: accept the race.** No install token or additional gate. "Complete setup immediately after install" is the final install step. Documented as a known accepted risk.
- **Web-enforced redirect, not daemon-enforced gate.** `GET /api/auth/state` returns `{ owner_exists, setup_complete, trusted_origins[] }`. Web middleware redirects: unauthenticated → login, no-owner → setup, setup-incomplete → setup. Daemon does not know about setup wizard progress.
- **Auto-persist setup origin; authenticated trust-on-first-visit for new origins.** Origin used during setup is auto-persisted to `trusted-origins.json`. Subsequent new origins (e.g. Tailscale IP) trigger a dismissible in-app banner offering one-click trust. Safe because `SameSite=Strict` ensures the JWT cookie is present only for direct navigation — third-party CSRF is not possible.
- **Direct-mode acknowledgement is a soft dismissible banner in Config.** No hard gate in setup. Phase 29 lands before any real user touches this. One-time dismissible; three accepted states persisted to daemon.
- **30-day absolute session expiry.** Silent redirect to login on expiration. No sliding window. Appliance-appropriate; low friction.
- **No migration code.** Existing P27 installs are treated as first-run for P28. No auth state to migrate.
- **CSRF enforcement lives in the web layer.** Daemon owns storage and auth endpoints; web enforces origin validation. Split at the daemon/web seam.
- **`session-secret` is generated by the daemon on first startup.** Detect-missing, generate, write. Same create-if-absent pattern as P27.02 daemon bootstrap.
- **Web reads `session-secret` from the filesystem at startup** alongside the bearer token in `hooks.server.ts`. No daemon HTTP endpoint for secret material.
- **Shell-level auth guard is sufficient for destructive action gating.** JWT middleware on the web layer blocks all unauthenticated access to the app shell. No per-action session checks needed.

## Exit Condition

On DS918+ DSM 7.1: first visit creates owner account; unauthenticated user sees no app shell or diagnostics; login and logout work; session survives navigation; destructive actions are blocked when logged out; LAN and Tailscale origins both reach full functionality; direct-mode acknowledgement state is persisted and visible. Pirate Claw web is no longer an unauthenticated control surface.

## Review Rules

Review and merge in ticket order.

Do not start the next ticket until:

- tests/checks for the current ticket pass
- ticket rationale is updated for behavior/tradeoff changes
- the unauthenticated surface contract (no diagnostics, no app shell, no destructive controls) is preserved

**Dependency gate after P28.01:** P28.02 and P28.03 depend on daemon auth endpoints being merged. P28.02 may be developed against the P28.01 branch if needed, but must be rebased and validated against merged daemon state before its PR is opened.

## Explicit Deferrals

- OpenVPN bridge (Phase 29)
- WireGuard support (v2)
- multi-user roles
- SSO/OIDC
- HTTPS certificate management
- public internet support
- audit logs
- password reset via email or external identity
- trusted origin management UI beyond the trust-on-first-visit banner
- explicit CSRF token layer (deferred unless a gap is discovered; `SameSite=Strict` covers the v1 threat model)

## Stop Conditions

Pause for review if:

- daemon auth endpoints require changes that affect the P27.02 install bootstrap contract
- session-secret generation conflicts with the existing create-if-absent pattern
- trusted origin enforcement breaks LAN health check access for unauthenticated install health state

## Phase Closeout

- **Retrospective: required**
- **Why:** P28 introduces a durable security boundary — owner auth, session management, trusted origins — and is the first human-facing security surface in Pirate Claw. Will generate learning that shapes P29+ delivery assumptions.
- **Trigger:** architecture/process impact + durable-learning risk
- **Artifact:** `notes/public/phase-28-retrospective.md`
- **Scope:** the final docs ticket (P28.05) must include retrospective writing.
